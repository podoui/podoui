# PODO Design System — Figma 플러그인

PODO 디자인 시스템 원본 파일의 **로컬 변수(컬렉션·모드 포함), 로컬 스타일, 모든 컴포넌트 / 컴포넌트 세트**를
JSON 파일 하나로 내보내고, 다른(빈) 파일에서 `_podo` 페이지를 만들어 동일하게 복원하는 플러그인입니다.

추가로 **프로젝트로 보내기**를 지원합니다: 로컬에서 `npx podoui`(또는 `podo import`)가
대기 중이면 현재 파일의 변수·스타일·컴포넌트를 `http://127.0.0.1:4141-4145`로 전송해
설치 프로젝트의 `.podo` 스펙으로 변환합니다. 네트워크 허용 범위는 이 로컬 루프백
포트 목록뿐입니다(`manifest.json`의 `networkAccess.allowedDomains`).

## 요구 사항

- **Figma 데스크톱 앱** — 개발(로컬) 플러그인은 데스크톱 앱에서만 로드할 수 있습니다.
- **원본 PODO 파일 편집 권한** — 내보내기는 원본 파일 안에서 플러그인을 실행해야 합니다.
- **폰트 설치**: Pretendard, IBM Plex Mono, SF Mono, Audiowide
  - 폰트가 없어도 가져오기는 중단되지 않습니다. 해당 폰트를 쓰는 텍스트가 다른 폰트로 대체될 수
    있다는 **경고**가 리포트에 남습니다. (특히 SF Mono는 macOS 외 환경에 없을 수 있습니다.)

## 개발 플러그인 로드 방법

```bash
cd figma-plugin
npm install
npm run build   # dist/code.js, dist/ui.html 생성
```

1. Figma 데스크톱 앱에서 아무 파일이나 엽니다.
2. 메뉴 → **Plugins → Development → Import plugin from manifest…**
3. 이 폴더의 `manifest.json`을 선택합니다.
4. 이후 **Plugins → Development → PODO Design System**으로 실행합니다.

## 사용 방법

### 1단계 — 내보내기 (원본 PODO 파일에서)

1. 원본 *PODO Design System* 파일을 열고 플러그인을 실행합니다.
2. **JSON 내보내기** 버튼을 클릭합니다.
3. 진행 표시(페이지 로드 → 변수 → 스타일 → 컴포넌트 직렬화)가 끝나면
   `<파일명>.podo-export.json` 파일이 다운로드됩니다. 컴포넌트가 800개 이상이라
   파일이 수십 MB에 이를 수 있습니다.

내보내기 범위는 **변수 전체 + `Component` 페이지**입니다
(`src/export/index.ts`의 `EXPORT_PAGE_NAMES`로 조정). 다른 페이지나 외부
라이브러리의 컴포넌트라도 **인스턴스로 참조되는 것은 자동으로 함께 내장**되어
가져오기 시 `External` 섹션에 로컬 컴포넌트로 복원됩니다 — 라이브러리 접근
권한이 필요 없습니다.

### 프로젝트로 보내기 (Figma → 로컬 프로젝트 `.podo`)

1. 로컬 프로젝트 터미널에서 `npx podoui` → **1) 피그마에서 가져오기** (또는 `podo import`)를
   실행해 수신 대기 상태로 둡니다. 기본 포트는 4141이고, 사용 중이면 4142~4145로
   자동 이동하며 터미널에 실제 포트가 표시됩니다.
2. 피그마에서 플러그인을 열고 **프로젝트로 보내기**를 누릅니다. 포트가 4141이 아니면
   터미널에 표시된 포트로 바꿔 입력합니다.
3. 터미널에 변환 계획(생성/갱신 파일, 경고, 충돌)이 표시되고, 승인하면 `.podo` 아래에
   토큰/테마/컴포넌트 스펙이 쓰입니다. 이후 `podo build`로 반영합니다.

### 2단계 — 가져오기 (빈 대상 파일에서)

1. 빈 파일(정확히는 `_podo` 페이지가 **없는** 파일)을 열고 플러그인을 실행합니다.
2. 파일 선택에서 1단계의 JSON을 고른 뒤 **가져오기 실행**을 클릭합니다.
3. `_podo` 페이지가 생성되고 **변수 → 스타일 → 컴포넌트** 순서로 복원됩니다.
   원본 페이지마다 같은 이름의 SECTION이 만들어지고 그 안에 컴포넌트가 배치됩니다.
4. 완료되면 생성된 개수(변수 컬렉션/변수/스타일/컴포넌트 세트/컴포넌트)와
   경고 목록이 리포트로 표시됩니다.

`_podo` 페이지가 이미 있으면 가져오기가 즉시 중단됩니다. 기존 `_podo` 페이지를
삭제하거나 이름을 바꾼 뒤 다시 실행하세요.

### (선택) 스냅샷 내장 — 내보내기 단계 없이 배포하기

내보내기 JSON을 플러그인 번들에 내장하면, 받는 사람은 빈 파일에서 버튼 한 번으로
`_podo`를 생성할 수 있습니다 (JSON 파일 주고받기 불필요).

1. 1단계에서 내보낸 JSON을 `figma-plugin/snapshot.json`으로 저장합니다.
2. `npm run build` — 빌드가 스냅샷을 `dist/ui.html`에 내장합니다.
3. 플러그인 UI에 **내장 스냅샷 가져오기** 섹션이 나타납니다 (원본 파일명·내보낸 날짜 표시).

원본 디자인 시스템이 바뀌면 내보내기를 다시 실행해 `snapshot.json`을 갱신하고
rebuild 하면 됩니다. `snapshot.json`은 git에 커밋되지 않습니다(.gitignore).

## 제약사항

- **변수 컬렉션의 표시 순서는 복제되지 않습니다.** Figma는 컬렉션을 기본
  알파벳순으로 나열하고, 원본의 순서(theme부터)는 UI 수동 정렬("Reorder
  collections") 결과인데 이는 플러그인 API에 노출되지 않습니다. 설치 후 변수
  모달 사이드바의 **⋯ → Reorder collections**에서 한 번 정렬하세요. 단,
  재설치(덮어쓰기)는 컬렉션을 지우고 다시 만들므로 수동 순서가 초기화될 수
  있습니다.
- **변수 모드 개수는 대상 파일의 플랜 제한을 따릅니다.** Starter 파일은 컬렉션당
  모드 1개만 허용되어 `theme`의 dark 등 두 번째 이후 모드가 드롭되고 경고가 남습니다
  ("could not add mode …"). light/dark를 모두 가져오려면 **Professional 이상
  팀(예: org 팀 스페이스)에 만든 파일**에서 가져오기를 실행하세요.
- **외부 라이브러리 인스턴스**: 참조된 외부 컴포넌트는 export JSON에 내장되어 로컬로
  복원됩니다. 내장 직렬화가 실패한 경우에만 `importComponentByKeyAsync`(라이브러리
  접근 권한 필요)로 폴백하고, 그마저 안 되면 `⚠ missing: <이름>` placeholder 프레임을
  만들고 경고를 남깁니다.
- **프로토타입 reactions**: best-effort로 복원합니다. 대상 노드가 복원 범위 밖이면
  해당 reaction은 생략되고 경고가 남습니다.
- **폰트 미설치**: 경고 후 계속 진행합니다 (위 요구 사항 참고).
- **SLOT 컴포넌트 속성**: 플러그인 API 지원 범위 안에서 best-effort로 복원하며,
  재현이 불가능한 경우 슬롯 내용을 일반 자식 노드로 복원하고 경고를 남깁니다.
- **텍스트 OpenType 세부 기능**(openTypeFeatures)은 플러그인 API에 setter가 없어
  복원되지 않습니다.
- 재생성된 에셋의 **라이브러리 게시/연결**(publish)은 다루지 않습니다.
- `_podo` 외의 페이지, 컴포넌트가 아닌 일반 캔버스 콘텐츠(문서용 프레임 등)는
  복제 대상이 아닙니다.
- 대상 파일이 Starter 플랜의 3페이지 제한에 걸려 있으면 페이지 생성 단계에서 실패합니다.

## 빌드 / 개발 명령

```bash
npm run build      # esbuild 번들 → dist/code.js + dist/ui.html 복사
npm run watch      # 변경 감지 재빌드
npm run typecheck  # tsc --noEmit (strict)
```
