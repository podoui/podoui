## Blocking findings

1. The 13 public component exports are `Button`, `Checkbox`, `Radio`, `Chip`, `Badge`, `Select`, `Input`, `Textarea`, `Field`, `Icon`, `Switch`, `Toast`, and `Tooltip`. Their top-level versions are not validly proven—and appear broken—in actual React Native. They are created with literal `"Pressable"`, `"Text"`, `"TextInput"`, and `"View"` host names ([index.tsx:326](/Users/tarucy/project/podoui/packages/native/src/index.tsx:326), [index.tsx:1385](/Users/tarucy/project/podoui/packages/native/src/index.tsx:1385)). React Native registers real primitives under native configurations such as `RCTView`, while `Pressable` is a composite component, not a registered host string. The native docs nevertheless recommend direct top-level imports ([installation-guide.md:87](/Users/tarucy/project/podoui/docs/installation-guide.md:87)). Report item 14 only renders unknown tags through React DOM, records inert `onPress` (`convClicks:0`), and the test suppresses all resulting React errors ([index.test.tsx:552](/Users/tarucy/project/podoui/packages/native/src/index.test.tsx:552)). That does not establish that the documented public exports render or function in React Native.

2. Disabled `Input` and `Textarea` do not render their disabled state correctly and remain focusable in the RNW consumer. `Input` only sets `editable:false`; its wrapper has branches for invalid and read-only but none for disabled ([index.tsx:842](/Users/tarucy/project/podoui/packages/native/src/index.tsx:842), [index.tsx:862](/Users/tarucy/project/podoui/packages/native/src/index.tsx:862)). `Textarea` has the same omission ([index.tsx:902](/Users/tarucy/project/podoui/packages/native/src/index.tsx:902), [index.tsx:921](/Users/tarucy/project/podoui/packages/native/src/index.tsx:921)). Consequently they retain normal background, border, and text colors instead of the established gray disabled treatment. Report items 9–11 prove only that typing is blocked and explicitly show `disabledAttr:false`; RNW therefore exposes a focusable read-only control rather than a disabled control. The `g10` screenshot even shows the Field-disabled input with a blue focus outline. No disabled computed-style or Tab-order assertion exists.

3. Native `Select` violates its published accessibility and keyboard contract. The JSON source requires Enter/Space opening, arrow navigation, Escape closing, focus remaining on the trigger with `aria-activedescendant`, and listbox/option semantics ([select.component.json:195](/Users/tarucy/project/podoui/packages/spec/samples/components/select.component.json:195)). Instead, every option is an independent `accessibilityRole:"button"` Pressable ([index.tsx:720](/Users/tarucy/project/podoui/packages/native/src/index.tsx:720)), the menu is an untyped View ([index.tsx:827](/Users/tarucy/project/podoui/packages/native/src/index.tsx:827)), and there is no arrow/Escape/active-option handling. Report item 6 confirms Space does not open the trigger and all three cells have `tabindex=0`. Calling that an RNW implementation detail does not make it conform to the contract being claimed.

4. `Select` cannot handle the documented long-list behavior. The spec says the menu caps at ten rows and scrolls beyond that ([select.component.json:8](/Users/tarucy/project/podoui/packages/spec/samples/components/select.component.json:8)); the native target’s limitations do not waive scrolling ([select.component.json:188](/Users/tarucy/project/podoui/packages/spec/samples/components/select.component.json:188)). The native host exposes no `ScrollView` ([index.tsx:22](/Users/tarucy/project/podoui/packages/native/src/index.tsx:22)), and `selectMenu` has neither a height cap nor scrolling ([index.tsx:1569](/Users/tarucy/project/podoui/packages/native/src/index.tsx:1569)). The harness avoids this with only three fruit and four city options ([App.jsx:105](/private/tmp/claude-501/-Users-tarucy-project-podoui/885735ab-d9d3-4bc8-9008-5a94bf821edb/scratchpad/harness/native/src/App.jsx:105)).

5. `Field` destroys an existing child description relationship. `wireNativeControl` unconditionally replaces `accessibilityDescribedBy` with the Field-generated value ([index.tsx:1413](/Users/tarucy/project/podoui/packages/native/src/index.tsx:1413)). React and Hono explicitly merge the child’s existing descriptor with the Field descriptor ([React index.tsx:2264](/Users/tarucy/project/podoui/packages/react/src/index.tsx:2264), [Hono index.tsx:1131](/Users/tarucy/project/podoui/packages/hono/src/index.tsx:1131)). The harness only uses children without pre-existing descriptions, so its Field wiring evidence cannot detect this accessibility regression.

6. `Icon` is not reliably hidden from Android accessibility. It only passes `accessibilityElementsHidden:true` ([index.tsx:1029](/Users/tarucy/project/podoui/packages/native/src/index.tsx:1029)), an iOS-only property. The report itself acknowledges this limitation in its honesty section ([native.md:52](/private/tmp/claude-501/-Users-tarucy-project-podoui/885735ab-d9d3-4bc8-9008-5a94bf821edb/scratchpad/reports/native.md:52)). No `importantForAccessibility="no-hide-descendants"`, `aria-hidden`, or equivalent Android behavior is supplied, and the unit host does not assert the hiding prop. Decorative glyph text can therefore be announced on Android.

## Non-blocking suggestions

1. Add a real React Native renderer/device smoke test, rather than treating React DOM unknown-tag rendering as native coverage.
2. Add coverage for remaining variants such as Button `fill`, Radio `lg`, all Switch sizes, Tooltip `left`, controlled Field value changes, and long Select lists.
3. Keep intentionally failing convenience-export experiments outside the main harness so genuine console regressions start from a clean baseline.
