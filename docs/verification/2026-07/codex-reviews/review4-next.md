## Blocking findings

1. DatePicker can emit a boundary time that its UI cannot render. `clampTimeToLimits` returns an exact non-step-aligned minimum such as `23:59` or `23:45` when no aligned value fits ([datepicker.tsx](/Users/tarucy/project/podoui/packages/react/src/datepicker.tsx:373)), but the minute `<select>` only contains step-aligned options and displays such values by flooring them—both become `23:30` ([datepicker.tsx](/Users/tarucy/project/podoui/packages/react/src/datepicker.tsx:1519)). That option is also disabled when below the minimum. The report’s B4 evidence rests only on emitted-value unit tests and expressly says these cases were not exercised through the harness UI (`next.md:64`), so it misses a real render/state mismatch.

2. Searchable Select loses focus after selection or Escape. While open, focus moves from the trigger to a conditionally rendered search input ([index.tsx](/Users/tarucy/project/podoui/packages/react/src/index.tsx:1277)). `pick()` and Escape merely close the menu ([index.tsx](/Users/tarucy/project/podoui/packages/react/src/index.tsx:1047), [index.tsx](/Users/tarucy/project/podoui/packages/react/src/index.tsx:1223)); nothing refocuses the trigger when the focused input unmounts. This violates the documented focus contract ([select.component.json](/Users/tarucy/project/podoui/packages/spec/samples/components/select.component.json:195)). Report A12 checks the value and closed menu but never `document.activeElement` (`next.md:43`); Space, ArrowUp, and Escape are also absent from the current Select evidence.

3. Editor’s controlled code-view synchronization can ignore a legitimate external update. `lastEmittedRef` permanently retains the last code-view emission, and every later external value equal to that historical value is discarded ([useCodeView.ts](/Users/tarucy/project/podoui/packages/react/src/editor/hooks/useCodeView.ts:70)). Sequence: emit A in code view, externally set B, then externally set A; the final update is ignored and the textarea remains B. B7 only tests one external update to a novel value (`next.md:67`).

4. Editor validation has a timer race. Every successful validation schedules an uncancelled timeout that clears status after two seconds ([index.tsx](/Users/tarucy/project/podoui/packages/react/src/editor/index.tsx:103)). If valid content is followed by invalid content before that timeout expires, the stale success timeout clears the newer error. The harness tests invalid → valid, which cannot expose valid → invalid stale-timer behavior (`next.md:67`).

5. Editor’s table and color controls are not functionally accessible from the keyboard. Table-size cells are click-only `<div>` elements without role, `tabIndex`, or key handling ([index.tsx](/Users/tarucy/project/podoui/packages/react/src/editor/index.tsx:882)); color swatches are empty buttons without an accessible name ([index.tsx](/Users/tarucy/project/podoui/packages/react/src/editor/index.tsx:710)). A23 activates these through pointer/programmatic clicks and limits its accessibility assertion to the contenteditable textbox (`next.md:109`, `next.md:114`, `next.md:118`).

6. Table accepts disabled rows in `defaultSelected`. Initial state copies the array without filtering ([index.tsx](/Users/tarucy/project/podoui/packages/react/src/index.tsx:1931)); disabled rows are excluded only when the later `selectableRows` list is built ([index.tsx](/Users/tarucy/project/podoui/packages/react/src/index.tsx:1950)). Consequently, `defaultSelected={[disabledIndex]}` renders a disabled row as selected, contradicting disabled-row selection semantics. The harness uses only selectable row 0 as its default (`KitchenSink.jsx:129`), so A18 does not cover this case.
