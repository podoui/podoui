# Podo Editor Migration Review

---

## Verdict

**PARTIAL — 프로덕션 릴리스 불가**

v2는 에디터 코어 인프라(CRUD UI, 프리뷰, 토큰 그룹핑, LNB 스크롤)를 확보했으나, 컴포넌트 커버리지·테마 시스템·배포 파이프라인 세 축에서 구조적 공백이 존재한다. legacy fixtures가 단 3개 컴포넌트에 머물러 있고, JSON-spec 기반 테마 분기 로직과 설치 프로젝트 연동 플로우가 미구현 상태다. 이 상태로 old main을 대체하면 기존 소비자에게 기능 퇴보가 발생한다.

---

## Blocking Gaps

### 1. 컴포넌트 커버리지 심각한 미달

| 구분               | 항목                                                                                                                                                  |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| v2 커버            | Button, Field, Input (3종)                                                                                                                            |
| old main 대비 누락 | avatar, checkbox-radio, chip, datepicker, doc-tabs, editor, file, label, pagination, select, tab, table, textarea, toast, toggle, tooltip (16종 이상) |

legacy-fixtures.ts에 색상/간격/반경/타이포그래피 토큰은 있으나, 위 컴포넌트에 대한 JSON spec이 전무하다. **JSON-spec-first 요건의 핵심 대상이 누락된 것이므로 차단 이슈로 분류한다.**

### 2. 테마 시스템 미구현

- old main은 `light / dark / auto / warm` 4종 테마를 지원했지만, v2 수정 계획에서는 `warm`을 제외하고 `light / dark / auto`를 우선 지원한다.
- v2에는 토큰 타입·변형(variation) 그룹핑만 존재하며, 테마 분기를 위한 JSON spec 구조나 런타임 스위처가 확인되지 않는다.
- 요건상 "theme support"가 명시되어 있으므로, 테마 키 기반 토큰 오버라이드 레이어가 spec에 포함되지 않으면 소비 프로젝트가 테마를 적용할 수단이 없다.

### 3. 그리드 시스템 누락

- old main은 12 / 6 / 4 컬럼 그리드를 정의했다.
- v2 현황에서 그리드 관련 토큰·컴포넌트 spec의 언급이 없다.
- 레이아웃 토큰이 없으면 설치 프로젝트의 그리드 커스터마이징이 불가능하다.

### 4. Button spec 불완전

- old main의 Button은 `테마 × 변형(variant) × 크기(size) × 상태(state) × 정렬(alignment)` 5차원 조합을 지원했다.
- v2의 legacy-fixtures.ts가 이 조합 매트릭스를 전부 포괄하는지 확인되지 않는다. 부분 커버는 소비 프로젝트에서 런타임 오류로 이어진다.

### 5. 설치 프로젝트 커스터마이징 및 빌드/업데이트 플로우 미구현

- 요건에 "installed-project customization and build/update flow"가 명시되어 있다.
- 현재 v2에는 에디터 내 CRUD와 프리뷰만 있고, 생성된 spec을 소비 프로젝트에 설치하거나 업데이트를 전파하는 파이프라인(CLI, 패키지 배포, 토큰 diff 적용 등)에 대한 구현 또는 설계가 확인되지 않는다.
- 이 플로우 없이는 에디터가 독립 프로토타입에 머물며 실제 사용 가치가 없다.

### 6. spec-editing.ts 미완성 가능성

- 현재 `spec-editing.ts`가 untracked 상태(`??`)로 존재한다.
- 이 파일이 JSON-spec-first CRUD의 핵심 로직을 담는다면, 아직 커밋되지 않은 미완성 코드가 브랜치 밖에 있다는 의미이며 안정성을 보장할 수 없다.

---

## Recommended Fix Plan

아래는 우선순위 순서로 정렬한 수정 계획이다. 각 항목은 독립적으로 착수할 수 있으나, 1→2 순서 의존성은 지킬 것.

### P0 — 브랜치 안정화 (즉시)

1. `spec-editing.ts`와 `legacy-fixtures.ts`를 검토하여 프로덕션 포함 여부를 결정하고 커밋하거나 `.gitignore`에 명시적으로 제외한다.
2. `index.test.tsx` 변경 사항을 확인하여 기존 테스트가 회귀 없이 통과하는지 검증한다.

### P1 — 테마 시스템 설계 (1주 이내)

1. JSON spec에 `theme` 최상위 키를 정의한다 (`light | dark | auto`). `warm`은 v2 초기 수정 범위에서 제외한다.
2. 각 테마별 토큰 오버라이드 맵을 spec 구조에 포함시키고, 에디터 프리뷰에서 테마 전환이 가능하도록 한다.
3. 그리드(12/6/4) 토큰을 레이아웃 섹션으로 spec에 추가한다.

### P2 — 컴포넌트 spec 확장 (2~3주)

1. old main 컴포넌트 목록을 기준으로 16종에 대한 JSON spec 템플릿을 작성한다.
2. Button의 5차원 조합 매트릭스를 legacy-fixtures에서 spec으로 마이그레이션하고 누락 케이스를 보완한다.
3. 각 컴포넌트 spec에 대한 프리뷰 렌더러가 에디터에서 동작하는지 통합 테스트로 검증한다.

### P3 — 빌드/업데이트 플로우 구현 (3~4주)

1. 에디터에서 생성한 spec을 내보내는 export 인터페이스(JSON 파일 또는 패키지 tarball)를 정의한다.
2. 설치 프로젝트에서 spec을 consume하는 CLI 또는 SDK 레이어를 설계한다.
3. 토큰/컴포넌트 spec 변경 시 diff를 감지하고 설치 프로젝트에 업데이트를 적용하는 플로우를 구현한다.
4. 커스터마이징 레이어(설치 프로젝트가 base spec을 오버라이드하는 방식)를 문서화하고 에디터 UI에서 편집 가능하게 한다.

### P4 — 검증 및 릴리스 게이팅

- old main의 모든 컴포넌트·테마 조합을 커버하는 회귀 테스트 스위트를 작성한다.
- 설치 프로젝트 시뮬레이션 E2E 테스트를 CI에 포함한다.
- 위 P0~P3가 모두 완료되고 테스트가 통과된 시점에만 `v2 → main` 머지를 허용한다.

---

> **결론:** v2는 에디터 기반을 잘 놓았으나 현재 상태로는 old main의 기능 수준에 미치지 못한다. P0~P3를 순서대로 완료한 후 재검토를 권장한다.
